System Purpose
RERA is a structured real-estate risk evaluation engine designed to assess
development projects based on regulatory, legal, and environmental indicators.
The system produces standardized risk reports while enforcing billing
integrity, audit traceability, and operational review controls.

Core Architecture
- Django backend with PostgreSQL data layer (Railway)
- React + Vite frontend (Vercel)
- Immutable credit ledger for billing
- JWT authentication with short-lived access tokens
- Idempotent evaluation endpoint
- Structured audit logging system
- Request correlation via request_id
- Superadmin bypass layer for internal access without billing
- Public Contact Us intake surface with async email notification

Billing Model
The system operates on a credit-based billing model. Purchases create positive
ledger entries. Each evaluation deducts one credit through a usage entry.
Ledger records are append-only and cannot be modified or deleted in production.

Superadmin Bypass
Users with is_superuser = True bypass billing and credit enforcement for
report access, while still remaining fully visible in audit and admin flows.

Financial Integrity Guarantees
- No duplicate charges under request retries
- Idempotency-Key required for evaluation requests
- Replay requests return stored responses
- Payload conflicts produce HTTP 409
- Zero balance produces HTTP 402 Payment Required
- Row-level locking prevents concurrent double-deduction

Monetization Surface
Free (teaser):
- Total score
- Risk band
- Project name, city, region

Paid (full report):
- Category breakdown
- Risk signals
- Information gaps
- Due diligence suggestions

Access determined by:
- Active subscription
- Sufficient credit balance
- Manual payment approval path
- Superadmin bypass

Interview & Evaluation Pipeline
Interview Answers
-> Interview Scoring Engine (reports/interview_scoring.py)
-> Category Scores (5 categories)
-> Evaluation Engine (rera_core/engine_v1.py)
-> Risk Band Classification (reports/risk_band.py)
-> Explanation Engine (reports/explanation_engine.py)
-> Billing Deduction (billing/services.py)
-> Structured API Report

Interview API (JWT-authenticated)
POST   /api/v1/interview/start/
PATCH  /api/v1/interview/<id>/save/
GET    /api/v1/interview/<id>/
POST   /api/v1/interview/<id>/submit/

Report API
GET    /api/v1/reports/
GET    /api/v1/reports/<request_id>/

Billing API
GET    /api/v1/billing/credits/balance/
POST   /api/v1/billing/credits/purchase/initiate/
POST   /api/v1/billing/credits/purchase/confirm/
POST   /api/v1/billing/subscription/activate/

Contact API
POST   /api/v1/contact/

Contact System
- Contact submissions are persisted in ContactMessage
- Submissions are throttled to reduce abuse
- Each submission emits an audit event
- Admin notification email is delivered via Resend HTTP API
- Delivery is asynchronous so the API response is not blocked by email latency

Operational Surfaces
- Django admin for model inspection and staff operations
- Audit dashboard in frontend for operational visibility
- Admin payments page for manual payment review
- Read-only AuditEvent admin with filters, search, and ordered history

Frontend Architecture
- React 19, Vite 7.3.1, React Router 7, Tailwind CSS
- Vitest + React Testing Library for frontend regression coverage
- One-command QA runner: powershell -File qa.ps1
- Interview API calls carry JWT auth where required
- Teaser report delivered before authentication
- Authentication and billing triggered at unlock/buy actions
- Contact page includes auto-dismissing success/error notices
- Footer navigation includes Privacy Policy, Contact, Terms of Service

Production Runtime
- Backend served by Gunicorn on Railway
- Static files served through WhiteNoise in app-hosted production
- collectstatic runs at startup before Gunicorn
- Railway PostgreSQL consumed through DATABASE_URL
