Staging Deployment Infrastructure (Completed March 15-21, 2026)

Deployment Model Finalized
RERA now supports split-host deployment architecture:

1. Frontend Service
   - React/Vite static site
   - Deployable to any static hosting or CDN
   - Environment: VITE_BACKEND_ORIGIN points to backend API domain
   - Build artifact: frontend/dist/

2. Backend Service
   - Django + Gunicorn WSGI application
   - Deployable to Python-capable hosts (VPS, Render, Fly.io, Railway)
   - Runtime: Linux systemd service
   - Configuration: Environment variables with Railway DATABASE_URL support

3. Database Service
   - PostgreSQL hosted on Railway or managed provider
   - Connection via DATABASE_URL or traditional DB_* variables
   - Connection pooling and SSL support

4. Reverse Proxy
   - Nginx serves frontend and proxies backend
   - Option: Single host (nginx + backend) or multi-host (frontend CDN + separate backend)

Database Configuration Enhancement
- Added direct DATABASE_URL parsing (railway://, postgresql:// schemes)
- Maintained backward compatibility with DB_NAME, DB_USER, DB_PASSWORD, DB_HOST, DB_PORT
- SSL mode: Configurable via DB_SSLMODE (preferred: "require" for Railway)

Environment-Driven Settings
- RERA_ENV controls which settings module loads (dev, staging, prod)
- New staging module: config.settings.settings_stage
- Security settings (SSL redirect, HSTS, cookie security) environment-controlled
- CORS and CSRF origins specified via environment variables
