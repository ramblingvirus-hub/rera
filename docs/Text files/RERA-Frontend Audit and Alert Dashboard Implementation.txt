
RERA – Frontend Audit & Alert Dashboard Implementation
Date: March 26, 2026
Status: READY FOR IMPLEMENTATION

---

## OBJECTIVE

Implement a Superadmin-only Audit Dashboard in the frontend that provides:

1. Full visibility into system audit events
2. Real-time detection of critical failures
3. Immediate alerting for:

   * Refund-triggering events
   * User experience gaps
   * Security intrusion attempts

This dashboard is NOT analytics.

It is a forensic and operational control panel.

---

## ACCESS CONTROL (CRITICAL)

* Route: /admin/audit
* Access: Superadmin ONLY
* Enforced via:

  * Backend: IsAdminUser permission
  * Frontend: route guard using user.is_superuser

NO public exposure of audit data.

---

## DATA SOURCE

Primary Table: AuditEvent

Fields:

* id
* request_id
* event_type
* severity (INFO / WARNING / ERROR)
* created_at
* metadata (JSON – optional but recommended)

Audit events are immutable and correlated via request_id.

---

## API CONTRACT

GET /api/v1/admin/audit/

Query params:

* request_id (optional)
* user_id (optional)
* event_type (optional)
* severity (optional)
* limit (default 100–200)

Response:
[
{
id,
request_id,
event_type,
severity,
created_at,
metadata
}
]

---

## FRONTEND STRUCTURE

Page: AuditDashboard.jsx

Sections:

1. Alert Banner (CRITICAL)
2. Filters Panel
3. Event Table
4. Request Timeline Viewer (modal or side panel)

---

1. ALERT BANNER (TOP PRIORITY FEATURE)

---

Purpose:
Immediately surface critical issues without requiring filtering.

Display conditions:
Triggered when ANY of the following events appear:

REFUND-RELATED EVENTS:

* PAYMENT_FAILED
* PAYMENT_VERIFIED but credit not issued
* BILLING_CREDIT_DEDUCTED without EVALUATION_SUCCESS
* Duplicate billing anomaly

USER EXPERIENCE GAPS:

* EVALUATION_VALIDATION_FAIL
* TIMEOUT / delayed evaluation (>20s)
* Missing category input errors
* Report generation failure

SECURITY / INTRUSION:

* LOGIN_FAILED (multiple attempts)
* THROTTLE_TRIGGERED
* SUSPICIOUS_ACTIVITY

---

UI Behavior:

* Sticky banner at top
* Color-coded:

  RED → Financial risk / refund required
  ORANGE → UX issue
  YELLOW → Security warning

---

Example:

[ RED ALERT ]
Possible refund required:

* Request ID: abc-123
* Credit deducted but evaluation failed

[ ACTION BUTTON ]
→ View Timeline

---

Logic (Frontend):

Scan latest audit events (last N = 50–100)

If:

* BILLING_CREDIT_DEDUCTED exists
  AND
* NO EVALUATION_SUCCESS for same request_id

→ Trigger REFUND ALERT

---

2. FILTER PANEL

---

Inputs:

* Request ID (text)
* User ID (text)
* Event Type (dropdown)
* Severity (dropdown)
* Date range (optional)

Buttons:

* Search
* Reset

---

3. EVENT TABLE

---

Columns:

* Timestamp
* Event Type
* Severity
* Request ID
* User ID (if available)

Behavior:

* Sort: newest first
* Click row → open timeline view

---

Severity Color Mapping:

INFO → gray
WARNING → yellow
ERROR → red

---

4. REQUEST TIMELINE VIEW

---

Triggered by clicking a row.

Displays all events with same request_id.

Format:

Request ID: XXXXX

[10:01:01] EVALUATION_ATTEMPT
[10:01:02] BILLING_CREDIT_DEDUCTED
[10:01:03] EVALUATION_SUCCESS

OR

[10:01:01] EVALUATION_ATTEMPT
[10:01:02] BILLING_CREDIT_DEDUCTED
[10:01:05] ERROR (evaluation failed)

---

Purpose:

* Debug issues
* Validate billing correctness
* Investigate disputes

Action:

---

3. UX FAILURE DETECTION

Condition:

* EVALUATION_VALIDATION_FAIL > threshold
* Repeated failures per user

Action:
→ Show ORANGE alert

4. SECURITY ALERT

Condition:

* > 3 LOGIN_FAILED within short window
* THROTTLE_TRIGGERED spike

Action:
→ Show YELLOW alert

Reason:
Possible brute-force or scraping attempt.
